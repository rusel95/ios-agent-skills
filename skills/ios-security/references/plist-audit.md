# Info.plist & Entitlements Security Audit

## How to Use This Reference

Read this file for every audit. Info.plist and entitlements define the app's security posture at the platform level. Many findings here are high-confidence — the configuration either meets the requirement or it doesn't.

## Locating Plist Files

1. **Info.plist**: Check each target's build settings for `INFOPLIST_FILE`. May be at project root, inside target folder, or generated by build tools (SPM, Tuist).
2. **Entitlements**: Look for `.entitlements` files. Check build settings for `CODE_SIGN_ENTITLEMENTS`.
3. **Privacy manifest**: `PrivacyInfo.xcprivacy` — should be in the app bundle and each SDK target.

## App Transport Security (ATS)

### P1: NSAllowsArbitraryLoads — 🔴 CRITICAL

```xml
<!-- ❌ CRITICAL: All ATS protections disabled globally -->
<key>NSAppTransportSecurity</key>
<dict>
    <key>NSAllowsArbitraryLoads</key>
    <true/>
</dict>
```

**Required action**: Remove `NSAllowsArbitraryLoads` or set to `false`. Use per-domain exceptions only where justified.

### P2: Partial ATS Exceptions — 🟡 HIGH

```xml
<!-- ❌ HIGH: ATS disabled for WebViews or media -->
<key>NSAllowsArbitraryLoadsInWebContent</key>
<true/>
<key>NSAllowsArbitraryLoadsForMedia</key>
<true/>
```

**Required action**: Evaluate whether these exceptions are necessary. WebView exception may be acceptable for apps loading arbitrary web content, but media exception should use specific domain exceptions instead.

### P3: Minimum TLS Version Below 1.2 — 🟡 HIGH

```xml
<!-- ❌ HIGH: TLS 1.0/1.1 allowed — known vulnerabilities -->
<key>NSExceptionDomains</key>
<dict>
    <key>legacy-server.example.com</key>
    <dict>
        <key>NSExceptionMinimumTLSVersion</key>
        <string>TLSv1.0</string>
    </dict>
</dict>
```

**Required action**: Require TLS 1.2 minimum. If legacy server cannot be upgraded, document the risk and plan migration.

### P4: Valid Exception Domain Audit

For each domain in `NSExceptionDomains`:
- [ ] Is the exception documented and justified?
- [ ] Is the scope as narrow as possible (specific domain, not wildcard)?
- [ ] Does `NSIncludesSubdomains` need to be `true`?
- [ ] Is `NSExceptionMinimumTLSVersion` at least `TLSv1.2`?

## URL Schemes

### P5: Custom URL Scheme Registration — 🟢 MEDIUM

```xml
<key>CFBundleURLTypes</key>
<array>
    <dict>
        <key>CFBundleURLSchemes</key>
        <array>
            <string>myapp</string>
            <string>myapp-callback</string>
        </array>
    </dict>
</array>
```

**Cross-reference with code**: Verify `application(_:open:options:)` validates source and sanitizes input (see M4 in medium-low-patterns.md).

**Recommendation**: Prefer Universal Links over custom URL schemes.

### P6: Universal Links Configuration

```xml
<!-- Entitlements file -->
<key>com.apple.developer.associated-domains</key>
<array>
    <string>applinks:example.com</string>
</array>
```

**Verify**: `apple-app-site-association` file is hosted at `https://example.com/.well-known/apple-app-site-association` with correct app ID and path matching.

## Privacy & Permissions

### P7: Camera, Microphone, Location Usage Descriptions

All `NS*UsageDescription` keys must be present when the corresponding API is used. Missing descriptions cause App Store rejection and indicate undeclared data collection.

**Required keys** (check if corresponding APIs are used in code):
- `NSCameraUsageDescription`
- `NSMicrophoneUsageDescription`
- `NSLocationWhenInUseUsageDescription` / `NSLocationAlwaysUsageDescription`
- `NSPhotoLibraryUsageDescription`
- `NSContactsUsageDescription`
- `NSCalendarsUsageDescription`
- `NSHealthShareUsageDescription` / `NSHealthUpdateUsageDescription`
- `NSFaceIDUsageDescription` (required for Face ID usage)

### P8: Missing Privacy Manifest — 🔵 LOW

**Detection**: Check for `PrivacyInfo.xcprivacy` in the project. Required since Spring 2024 for apps using "required reason APIs" (UserDefaults, file timestamps, disk space, system boot time).

```xml
<!-- ✅ REFERENCE: PrivacyInfo.xcprivacy structure -->
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>NSPrivacyAccessedAPITypes</key>
    <array>
        <dict>
            <key>NSPrivacyAccessedAPIType</key>
            <string>NSPrivacyAccessedAPICategoryUserDefaults</string>
            <key>NSPrivacyAccessedAPITypeReasons</key>
            <array>
                <string>CA92.1</string>
            </array>
        </dict>
    </array>
</dict>
</plist>
```

## Entitlements Audit

### P9: Overly Broad Entitlements — 🔵 LOW

Check each entitlement against actual usage:

| Entitlement | Verify Usage |
|-------------|-------------|
| `com.apple.developer.associated-domains` | Universal Links implemented? |
| `com.apple.developer.healthkit` | HealthKit APIs used? |
| `aps-environment` | Push notifications implemented? |
| `com.apple.developer.in-app-payments` | Apple Pay implemented? |
| `keychain-access-groups` | Shared Keychain needed between apps? |
| `com.apple.security.application-groups` | App Groups data sharing used? |

**Risk**: Unused entitlements increase attack surface and may grant unneeded capabilities.

### P10: App Group Shared Data — 🟢 MEDIUM

```xml
<key>com.apple.security.application-groups</key>
<array>
    <string>group.com.example.myapp</string>
</array>
```

**Cross-reference with code**: Search for `UserDefaults(suiteName: "group.com.example.myapp")` and `FileManager.default.containerURL(forSecurityApplicationGroupIdentifier:)`. Verify shared data is encrypted before writing.

## Binary Protection Flags

These are checked via build settings, not plist, but are part of the configuration audit:

### P11: Position Independent Executable (PIE)

**Detection**: Build setting `GCC_GENERATE_POSITION_DEPENDENT_CODE` should be `NO` (default). PIE enables ASLR.

### P12: Automatic Reference Counting (ARC)

**Detection**: Build setting `CLANG_ENABLE_OBJC_ARC` should be `YES`. Non-ARC code is vulnerable to use-after-free and double-free bugs.

### P13: Stack Canary

**Detection**: Build setting `GCC_ENABLE_STACK_PROTECTION` should be set. `-fstack-protector-all` flag should be present in `OTHER_CFLAGS` for maximum protection.

## Export Compliance

### P14: ITSAppUsesNonExemptEncryption

```xml
<key>ITSAppUsesNonExemptEncryption</key>
<true/>  <!-- Requires encryption export compliance documentation -->
```

If the app uses AES, TLS (beyond HTTPS), or custom encryption, this must be `true` and export compliance documentation filed.

## Info.plist Audit Checklist

- [ ] Check `NSAllowsArbitraryLoads` — must be `false` or absent
- [ ] Audit each `NSExceptionDomains` entry for justification and TLS version
- [ ] Check `NSAllowsArbitraryLoadsInWebContent` and `NSAllowsArbitraryLoadsForMedia`
- [ ] List all `CFBundleURLSchemes` — cross-reference with URL handler validation
- [ ] Verify Universal Links entitlement matches hosted AASA file
- [ ] Check all `NS*UsageDescription` keys match actual API usage
- [ ] Verify `PrivacyInfo.xcprivacy` exists and declares required reason APIs
- [ ] Audit entitlements for unused capabilities
- [ ] Check App Group data sharing for encryption
- [ ] Verify PIE, ARC, and stack canary build settings
</detection_checklist>
